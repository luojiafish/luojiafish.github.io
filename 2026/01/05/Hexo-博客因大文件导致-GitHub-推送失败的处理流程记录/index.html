Hexo 博客因大文件导致 GitHub 推送失败的处理流程记录

一、问题背景

在 Hexo 博客项目中，将体积较大的 PDF 文件放入仓库并尝试推送至 GitHub 时，出现如下错误：

File xxx.pdf is larger than 100MB; this exceeds GitHub's file size limit

即使在本地已经删除该 PDF 文件，GitHub 仍然拒绝 push，问题无法通过简单删除文件解决。

二、问题分析

问题不在于当前工作区是否存在大文件，而在于 Git 的提交历史中仍然包含该大文件。

GitHub 在接收推送时，会检查整个提交历史中的对象大小，而不是只检查当前提交内容。因此，只要某个历史 commit 中曾经包含超过 100MB 的文件，推送就会失败。

这也是许多“重新删除文件仍然无法 push”的原因。

对于 Hexo 博客而言，还需要注意一个特殊点：
Hexo 使用 hexo-deployer-git 部署时，会在项目根目录生成一个独立的部署仓库 .deploy_git，该目录本身是一个单独的 Git 仓库，与 Hexo 源码仓库完全不同。

因此，很多操作如果在错误的仓库中执行（例如在主仓库中 reset Pages 的 commit），都会失败。

三、定位过程

  1. 明确当前 GitHub 报错属于“大文件历史残留”问题，而不是当前文件问题。
  2. 确认 Hexo 部署机制，意识到 .deploy_git 是独立的 Git 仓库。
  3. 在 .deploy_git 目录中执行 git log --oneline，查看该部署仓库的提交历史。
  4. 发现部署仓库中只包含自动生成的 Site updated 类提交，未发现异常大文件。
  5. 判断之前尝试 reset 的某个 commit ID 并不属于当前仓库，因此 reset 报错属于“在错误仓库中操作”。

五、结果确认

  3. 博客页面能够正常访问。

六、经验总结

  1. GitHub 的文件大小限制检查的是整个提交历史，而不是当前文件状态。
  2. 删除文件并不能解决历史中已经存在的大文件问题。
  3. Hexo 的部署仓库 .deploy_git 是一个独立 Git 仓库，相关 Git 操作必须在该目录中进行。
  4. GitHub Actions 页面显示的 commit 并不一定属于部署仓库，不能直接用于 reset。
  5. 对于 Hexo 博客，大文件（如 PDF）不应直接进入 Git 仓库，应采用外链或 Release 方式管理。
